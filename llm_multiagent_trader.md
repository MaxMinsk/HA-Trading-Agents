# LLM-мульти-агент на Claude (по мотивам TradingAgents): подробный разбор

> Углублённый разбор подхода D из `initial_investigate.md`:
> **AI-агент, построенный на больших языковых моделях (Claude), который
> анализирует рынок как команда специалистов и принимает торговые решения.**
>
> Дата: 2026-06-22

---

## 1. Идея в одном абзаце

Вместо того чтобы обучать числовую модель предсказывать цену, мы используем
**Claude как рассуждающего аналитика**. Он собирает разнородные данные (цены,
индикаторы, новости, настроения, ончейн), **рассуждает** над ними и выдаёт
структурированное решение: `buy / sell / hold` + размер позиции + уверенность +
обоснование. Сила подхода — не в предсказании цифр, а в **синтезе
качественной и количественной информации** и в **объяснимости** каждого
решения. Самый мощный вариант — не один агент, а **команда специализированных
агентов**, которая имитирует трейдинг-фирму и приходит к решению через дебаты и
риск-контроль.

> ⚠️ Напоминание из основного документа: это **не печатная машинка для денег**.
> LLM не «видит будущее». Подход даёт объяснимый, гибкий процесс принятия
> решений и хорош для исследования edge — но реальное преимущество надо
> доказывать честным бэктестом и paper trading, а не верить академическим
> графикам.

---

## 2. Почему мульти-агент, а не один агент

Исследования 2025–2026 (TradingAgents, LiveTradeBench, «When Agents Trade»)
устойчиво показывают: **команда специализированных агентов бьёт одиночного
агента и пассивный buy&hold** в бэктестах/лайв-бенчмарках. Причины:

- **Разделение труда** — один агент, который одновременно читает новости,
  считает индикаторы и оценивает риск, перегружен и «размазывает» внимание.
  Специалист на узкой задаче точнее.
- **Дебаты снижают однобокость** — отдельные «бык» и «медведь» вынуждают
  систему рассмотреть обе стороны, прежде чем решить.
- **Независимый риск-контроль** — отдельный агент с правом вето ловит
  импульсивные/абсурдные решения трейдера.
- **Объяснимость** — видно, кто что сказал и почему принято решение.

Минус — дороже и медленнее (несколько вызовов LLM на одно решение). Поэтому
режим — **низкочастотный** (раздел 7).

## 3. Архитектура (детально)

```
                         ┌──────────────────────────────┐
                         │      DATA LAYER (не LLM)       │
                         │ цены/индикаторы, новости,      │
                         │ sentiment, ончейн, funding/OI  │
                         └───────────────┬────────────────┘
                                         │  (контекст для агентов)
      ┌──────────────┬───────────────────┼───────────────────┐
      ▼              ▼                    ▼                   ▼
┌───────────┐ ┌───────────┐       ┌───────────┐       ┌───────────┐
│Technical  │ │Fundamental│       │Sentiment  │       │ On-chain  │
│analyst    │ │/news      │       │analyst    │       │ analyst   │
│(индикаторы)│ │analyst    │       │(соцсети)  │       │(потоки)   │
└─────┬─────┘ └─────┬─────┘       └─────┬─────┘       └─────┬─────┘
      └─────────────┴───────────┬───────┴───────────────────┘
                                ▼
                      ┌───────────────────┐
                      │   RESEARCH / DEBATE │  «бык» vs «медведь»
                      │  bull agent ⇄ bear  │  N раундов спора
                      └─────────┬───────────┘
                                ▼
                      ┌───────────────────┐
                      │   TRADER AGENT     │  агрегирует → черновик решения
                      │ action/size/conf   │  (structured output JSON)
                      └─────────┬───────────┘
                                ▼
                      ┌───────────────────┐
                      │  RISK MANAGER      │  проверка лимитов, право ВЕТО
                      │ (LLM + жёсткие     │  корректирует size / блокирует
                      │  правила-код)      │
                      └─────────┬───────────┘
                                ▼
                      ┌───────────────────┐
                      │  EXECUTION (код)   │  ордер на Binance + журнал
                      │  + RISK LAYER код  │  стоп, плечо, kill-switch
                      └───────────────────┘
```

Ключевой принцип: **LLM-агенты дают сигнал и обоснование; исполнение и жёсткие
лимиты — это обычный детерминированный код**, который LLM не может обойти.

## 4. Роли агентов (вход → выход)

| Агент | Вход (контекст) | Задача | Выход |
|---|---|---|---|
| **Technical analyst** | свечи, RSI/MACD/MA/ATR, уровни | оценить тех. картину | мнение + уверенность |
| **News/fundamental** | новости, листинги, события | оценить значимые события | мнение + уверенность |
| **Sentiment analyst** | агрегированный sentiment (X/Reddit), fear&greed | настроение толпы | мнение + уверенность |
| **On-chain analyst** (опц.) | потоки на биржи, OI, funding | оценить позиционирование | мнение + уверенность |
| **Bull researcher** | мнения аналитиков | аргументы ЗА покупку | тезисы |
| **Bear researcher** | мнения аналитиков | аргументы ПРОТИВ | тезисы |
| **Trader** | все мнения + дебаты + текущая позиция | принять решение | `{action, size, confidence, rationale}` |
| **Risk manager** | решение трейдера + лимиты/портфель | проверить, скорректировать, вето | финальное решение или блок |

Аналитики могут работать **параллельно** (независимы) → быстрее и можно
батчить. Дебаты и трейдер — последовательно.

## 5. Оркестрация

Два пути:

1. **Готовый каркас — TradingAgents** (LangGraph; поддерживает Anthropic/Claude).
   Быстрый старт, готовая структура ролей и дебатов. Хорош как референс.
2. **Свой оркестратор** на Anthropic SDK. Больше контроля, проще встроить свой
   риск-слой и Binance-исполнение. Рекомендую для продакшена — каркас понятен.

Технические опоры от Claude API, которые делают агента надёжным:

- **Structured output** (`output_config.format` / `messages.parse`) — каждый
  агент возвращает **валидный JSON по схеме** (action/size/confidence/rationale),
  а не свободный текст. Убирает парсинг и «болтовню».
- **Tool use** — агент сам вызывает инструменты: `get_price`, `get_news`,
  `get_orderbook`, и (в проде, осторожно) `place_order`. Опасные инструменты
  (ордера) должны проходить через подтверждение/риск-слой.
- **Adaptive thinking** (`thinking={"type":"adaptive"}` + `effort`) — даёт
  модели «подумать» над сложным решением; для трейдера/риск-менеджера это
  важно, для дешёвой классификации sentiment — нет.
- **Prompt caching** — системный промпт и правила у каждого агента стабильны;
  кэширование префикса режет стоимость входных токенов до ~0.1×. Критично при
  многоагентном цикле (см. раздел 7).
- **Batch API** — если гоняем агентов по многим монетам/историческим точкам
  (бэктест) — батч даёт −50% к стоимости (асинхронно, до 24ч).

## 6. Формат решения (структурированный вывод)

Жёсткая схема финального решения — основа надёжности. Пример (Pydantic):

```python
from enum import Enum
from pydantic import BaseModel, Field

class Action(str, Enum):
    buy = "buy"
    sell = "sell"
    hold = "hold"

class TradeDecision(BaseModel):
    action: Action
    size_pct: float = Field(ge=0, le=1, description="доля капитала, 0..1")
    confidence: float = Field(ge=0, le=1, description="уверенность 0..1")
    rationale: str = Field(description="краткое обоснование")
    key_risks: list[str] = Field(description="главные риски решения")
```

Каждый аналитик возвращает похожую мини-схему (`view`, `confidence`,
`reasons`). Risk manager — `{approved: bool, adjusted_size_pct, veto_reason}`.

## 7. Частота и стоимость (важно — считаем заранее)

Цены Claude API (за 1M токенов, на 2026-06):

| Модель | ID | Вход $/1M | Выход $/1M | Роль в системе |
|---|---|---|---|---|
| Opus 4.8 | `claude-opus-4-8` | $5 | $25 | трейдер, риск-менеджер (сложное рассуждение) |
| Sonnet 4.6 | `claude-sonnet-4-6` | $3 | $15 | аналитики, дебаты (баланс цена/качество) |
| Haiku 4.5 | `claude-haiku-4-5` | $1 | $5 | дешёвая классификация sentiment, разметка новостей |
| Fable 5 | `claude-fable-5` | $10 | $50 | самые тяжёлые решения (опционально) |

> По умолчанию для ответственных решений бери Opus 4.8; Sonnet/Haiku — это
> осознанная оптимизация стоимости там, где задача проще. Выбор модели —
> твоё решение; здесь — ориентир.

**Грубый расчёт одного решения (мульти-агент, ~6 вызовов):**
- допустим, на цикл уходит ~40K входных + ~6K выходных токенов суммарно;
- без кэша на смеси Sonnet/Opus это ≈ $0.1–0.3 за решение;
- с **prompt caching** входные токены дешевеют ~в 10× → ≈ $0.03–0.1.

**Вывод по частоте:**
- **Скальпинг/HFT — невозможно** (дорого и медленно: каждый «тик
  рассуждения» = вызовы API на секунды и центы).
- **Реалистичный режим — низкочастотный:** решение раз в **1–4 часа** или раз
  в день, на свинг/дневном горизонте. Тогда стоимость API — копейки против
  потенциального оборота, а латентность не критична.

## 8. Бэктестинг LLM-агентов (отдельная боль)

Бэктест LLM-агента сложнее, чем числовой модели, из-за двух ловушек:

1. **Look-ahead через знания модели.** Если прогонять агента на старых данных,
   модель может «знать» исход (что проект рухнул, что был памп). Это рисует
   ложно хороший бэктест.
   - **Лечение:** строго подавать только **point-in-time** данные (новости/цены
     по состоянию на момент решения), тестировать на периодах **после
     обучающего среза** модели, либо ограничивать агенту доступ к информации
     «из будущего» через инструменты.
2. **Стоимость и время бэктеста.** Каждая историческая точка = несколько
   вызовов LLM. Прогон на годах истории дорог.
   - **Лечение:** **Batch API** (−50%), prompt caching, прореживание точек
     (раз в день), сначала короткий период для отладки.

Остальное — как в основном документе: учёт комиссий/слиппеджа, walk-forward,
честный out-of-sample, затем paper trading на testnet.

---

## 9. Риск-слой (детерминированный, поверх агентов)

LLM может «уверенно ошибаться». Поэтому **последнее слово — у кода**:

- Жёсткие лимиты: max доля капитала на сделку, max плечо (низкое!), max позиций.
- Sanity-проверки сигнала: отклонять абсурд (например size > лимита,
  противоречие позиции).
- Дневной лимит убытка → авто-стоп бота.
- Стоп-лоссы/тейк-профиты на бирже (не «в голове агента»).
- Kill-switch (ручной + авто) — закрыть всё при аномалии/сбое API.
- Контроль деградации: если live-метрики расходятся с бэктестом — стоп.
- Безопасность: API-ключи **без права вывода**, привязка к IP, субаккаунт.

Risk-manager-агент — это *первый* фильтр (умный, объясняет), но за ним стоит
*второй* фильтр — тупой и непреклонный код.

## 10. Сильные и слабые стороны (честно)

**Сильные:**
- Сочетает количественное и качественное (новости/нарративы) — то, в чём
  числовые модели слабы.
- Объяснимость: каждое решение с обоснованием → легко аудировать и улучшать.
- Быстрый прототип: не нужно обучать модель, нужен хороший промптинг + данные.
- Идеально ложится на разработку через Claude.

**Слабые / риски:**
- Стоимость и латентность → только низкочастотный режим.
- Галлюцинации/непостоянство → нужны structured output, ограничители, риск-вето.
- Нет врождённого предсказательного edge по цене — сила в синтезе, и доходность
  спорна.
- Look-ahead в бэктесте — методологически коварно.
- Зависимость от качества данных (особенно новостей/sentiment в реальном времени).

## 11. MVP — пошаговый план

**Этап 0 — данные.** Модуль, который на любой момент времени собирает:
цены/индикаторы + свежие новости + sentiment (+ funding/OI). Строго
point-in-time. Это 80% работы.

**Этап 1 — один агент.** Сначала **single-agent**: один Claude-вызов со
structured output, который по контексту выдаёт `TradeDecision`. Прогнать
бэктест/paper. Это baseline — мульти-агент должен его побить.

**Этап 2 — мульти-агент.** Добавить аналитиков (параллельно) → дебаты →
трейдер → риск-менеджер. Сравнить с baseline честно.

**Этап 3 — риск-слой + исполнение.** Детерминированные лимиты, kill-switch,
ордера на Binance testnet.

**Этап 4 — paper trading** на testnet ≥ 2–4 недели, сверка P&L.

**Этап 5 — live** минимальной суммой, Telegram-алерты, журнал решений (с
обоснованиями агентов), еженедельный разбор.

## 12. Пример кода (упрощённый single-agent, отправная точка)

```python
import anthropic
from pydantic import BaseModel, Field
from enum import Enum

client = anthropic.Anthropic()  # ключ из ANTHROPIC_API_KEY

class Action(str, Enum):
    buy = "buy"; sell = "sell"; hold = "hold"

class TradeDecision(BaseModel):
    action: Action
    size_pct: float = Field(ge=0, le=1)
    confidence: float = Field(ge=0, le=1)
    rationale: str
    key_risks: list[str]

SYSTEM = """Ты — дисциплинированный крипто-трейдер на дневном горизонте.
Ты НЕ предсказываешь будущее с уверенностью; ты взвешиваешь вероятности и риск.
Решай консервативно: при неопределённости предпочитай hold или малый size.
Всегда учитывай комиссии и риск ликвидации. Возвращай только структуру решения."""

def decide(context: str) -> TradeDecision:
    resp = client.messages.parse(
        model="claude-opus-4-8",
        max_tokens=2000,
        thinking={"type": "adaptive"},          # дать модели подумать
        output_config={"effort": "high"},
        system=[{                                # кэшируем стабильный префикс
            "type": "text",
            "text": SYSTEM,
            "cache_control": {"type": "ephemeral"},
        }],
        messages=[{"role": "user", "content": context}],
        output_format=TradeDecision,            # строгий JSON по схеме
    )
    return resp.parsed_output

# context — собранный point-in-time снимок: цена, индикаторы, новости,
# sentiment, текущая позиция и баланс.
decision = decide(context="...")
print(decision.action, decision.size_pct, decision.confidence)
print(decision.rationale)
# дальше: deterministic risk layer -> (testnet) ордер
```

Для мульти-агента: те же вызовы для аналитиков (можно `claude-haiku-4-5` /
`claude-sonnet-4-6`), затем агрегирующий вызов трейдера (`claude-opus-4-8`),
затем риск-менеджер. Аналитиков по разным монетам/точкам можно гнать через
Batch API для экономии.

## 13. Реалистичные ожидания

| Ожидание | Реальность |
|---|---|
| «Команда Claude-агентов = стабильная прибыль» | ❌ Нет гарантии; edge надо доказать |
| «Можно скальпить на LLM» | ❌ Дорого/медленно; только низкочастотно |
| «Академические +150% повторятся у меня» | ❌ Publication bias, зависят от периода |
| «Объяснимый, гибкий, современный AI-агент как проект» | ✅ Да — отличная цель |
| «Хорошо сочетает новости + цифры» | ✅ Главная сила подхода |

---

## 14. Итог

LLM-мульти-агент на Claude — **самый современный и наглядный «AI-агент»** под
твою задачу: команда специалистов спорит и решает, риск-менеджер контролирует,
детерминированный код исполняет и страхует. Работает в **низкочастотном
режиме** (свинг/день), отлично сочетает количественные и качественные данные и
полностью объясним. Но к доходности относись как исследователь: сначала
baseline (один агент), потом мульти-агент, строгий бэктест против look-ahead,
testnet, минимальный капитал — и только потом масштабирование.

> Следующий шаг: собрать **point-in-time data layer** (раздел 11, этап 0) —
> без него ни single-, ни мульти-агент честно не протестировать.
